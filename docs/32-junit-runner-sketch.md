## 1. Problem

`lyric test` for the .NET target works by compiling each `@test_module` file
through the self-hosted `Lyric.TestSynth` rewriter, producing a `main()` that
runs assertions and emits TAP-shaped output.  The F# test runner then executes
the resulting binary.

For the JVM target this flow breaks down:

1. The synthesised `main()` approach does not integrate with standard JVM
   tooling (IDEs, Gradle, Maven Surefire).
2. IDE test-gutter icons rely on JUnit 5's `TestEngine` discovery protocol.
3. Checked-exception wrapping (Q-J012/Q-J013) and Maven-linked dependencies
   (D053) need test runs to exercise live JVM FFI.

The existing `Lyric.TestSynth` binary still works for quick CLI smoke tests
but is not sufficient for full JVM test integration.

## 2. Design goals

- **No new runtime dependency for Lyric projects** â€” the adapter is part of
  the Lyric toolchain JAR, not something projects must declare.
- **Piggy-back on JUnit 5** â€” use `TestEngine` / `LauncherFactory` rather
  than a bespoke runner (avoids reinventing discovery, filtering, reporting).
- **Single annotation** â€” a `@LyricTest` retention annotation emitted by the
  compiler bridges the Lyric `@test_module` / `ITest` world with JUnit 5.
- **IDE-compatible** â€” test classes visible to the standard JUnit 5 platform
  launcher so VS Code / IntelliJ gutter icons work.

## 3. `@LyricTest` annotation

The JVM emitter emits a `@lyric.runtime.jvm.LyricTest` annotation on each
synthesised test wrapper method.  The annotation is `@Retention(RUNTIME)` and
`@Target(METHOD)` so that the `LyricTestEngine` can discover it via reflection.

```java
// generated by Lyric compiler, not hand-authored
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface LyricTest {
    String displayName() default "";
    String sourceFile()  default "";
    int    sourceLine()  default 0;
}
```

`displayName` is taken from the `ITest` item's doc comment or function name.
`sourceFile` / `sourceLine` carry the span for IDE navigation.

## 4. Emitted class shape

For a `@test_module` file `math_tests.l` containing three `@test` functions,
the emitter produces a single class `math_tests` with:

```java
// synthesised â€” not in the Lyric source
public final class math_tests {

    @LyricTest(displayName = "addition works", sourceFile = "math_tests.l", sourceLine = 7)
    public static void __lyric_test_0() { /* ... */ }

    @LyricTest(displayName = "subtraction is inverse of addition", ...)
    public static void __lyric_test_1() { /* ... */ }

    @LyricTest(displayName = "multiply by zero", ...)
    public static void __lyric_test_2() { /* ... */ }
}
```

Each method:
- Is `public static void`.
- Throws on assertion failure by converting a Lyric `@test` assertion error to
  a `java.lang.AssertionError` (wrapping the Lyric message string).
- Is discovered by the `LyricTestEngine` via annotation scanning.

The existing `Lyric.TestSynth` rewriter produces the same function naming
(`__lyric_test_<i>`); the JVM emitter adds the `@LyricTest` annotation and
the `AssertionError` throw conversion.

## 5. `LyricTestEngine`

The Lyric toolchain ships a `LyricTestEngine` implementing
`org.junit.platform.engine.TestEngine`.  It is registered via the standard
ServiceLoader mechanism (`META-INF/services/â€¦`).

### 5.1 Discovery

```
LyricTestEngine.discover(request)
  for each class on the test classpath:
    for each public static void method annotated with @LyricTest:
      emit TestDescriptor(uniqueId, displayName)
```

### 5.2 Execution

```
LyricTestEngine.execute(request)
  for each TestDescriptor in the plan:
    try:
      method.invoke(null)     // static; no receiver
      notify SUCCEEDED
    catch AssertionError e:
      notify FAILED(e.message)
    catch Throwable t:
      notify FAILED("unexpected exception: " + t)
```

No reflection tricks are needed beyond standard `java.lang.reflect.Method`
invocation because the methods are `public static`.

### 5.3 Service registration

The Lyric toolchain JAR ships:

```
META-INF/services/org.junit.platform.engine.TestEngine
  lyric.runtime.jvm.LyricTestEngine
```

This makes `LyricTestEngine` auto-registered when the Lyric toolchain JAR is
on the test classpath alongside JUnit 5 platform.

## 6. `lyric test --jvm` flow

```
lyric test --jvm [file.l | dir/]
  1. Compile each @test_module file â†’ JVM class (via lyric build --target=jvm)
  2. Build a test classpath:
       - compiled test classes
       - Lyric stdlib JAR (JVM kernel variant)
       - Lyric toolchain JAR (contains LyricTestEngine)
       - JUnit 5 platform launcher + JUnit vintage engine JARs
         (resolved from the toolchain's bundled versions)
  3. Shell to:
       java -cp <classpath> \
            org.junit.platform.console.standalone.ConsoleLauncher \
            --select-class=<each test class> \
            --details=tree
  4. Forward exit code and TAP/XML output to the caller.
```

Step 2 bundles the JUnit 5 console standalone JAR inside the Lyric toolchain
distribution so projects do not need to declare it.

## 7. Maven project integration

For projects using `[maven]` dependencies, the same test classpath includes
restored Maven JARs.  The `lyric restore` step populates a lockfile cache
(per D053), and `lyric test --jvm` reads that cache to assemble the classpath.

```toml
# lyric.toml
[project]
name    = "my-service"
target  = "jvm"

[maven]
"org.junit.jupiter:junit-jupiter-api" = "5.10.2"
```

When `junit-jupiter-api` is a declared Maven dependency, Lyric test methods
can also use JUnit Jupiter assertions directly, and the JUnit 5 vintage engine
will co-discover both `@LyricTest` and `@Test`-annotated methods in the same
run.

## 8. Interaction with Q-J012 / Q-J013

Tests that exercise Java FFI (Maven-linked classes with checked exceptions)
require the JVM emitter call-site wrapping from Q-J013.  Until Q-J013 lands:

- Test methods that call checked-exception Java APIs must wrap the call site
  in `Std.Jvm.catch` (Q-J012).
- The `LyricTestEngine` catches `Throwable`, so unhandled checked exceptions
  from incomplete Q-J013 call sites surface as `FAILED` test results rather
  than silently swallowed.
